# 狼人杀游戏Web应用开发计划

## 目标描述
开发一个支持真人与AI混合对战的狼人杀游戏Web应用，用于录制真人AI狼人杀对战视频。系统需要支持6-16人灵活配置，1-4名真人玩家通过手机或电脑端参与，其余由AI玩家填充。游戏采用本地部署方式，通过局域网连接各设备。系统需集成6种大模型（GPT、Claude、Gemini、Deepseek、Qwen、Kimi）作为AI玩家，并实现完整的语音交互系统，包括语音识别、语音合成和游戏流程自动主持功能。

## 验收标准

遵循TDD理念，每个标准包含正向测试（预期通过）和反向测试（预期失败）以实现确定性验证。

- AC-1: 游戏规则系统完整实现
  - 正向测试（预期通过）：
    - 创建6人局（2狼2民2神）、9人局（3狼3民3神）、12人局（4狼4民4神）、16人局（5狼6民5神），游戏能正常开始并按规则运行
    - 预言家夜间验人、女巫使用解药/毒药、猎人死亡开枪、守卫守护玩家等技能按规则生效
    - 狼人阵营屠边（杀光所有神职或平民）时判定狼人胜利，好人阵营放逐所有狼人时判定好人胜利
  - 反向测试（预期失败）：
    - 创建5人局或17人局应提示人数不在6-16范围内
    - 女巫尝试对同一玩家连续两晚使用解药应被拒绝
    - 守卫连续两晚守护同一玩家应被拒绝
    - 游戏未结束时尝试开始新局应被阻止
  - AC-1.1: 规则介绍和教程系统
    - 正向测试：访问规则页面能看到完整的角色介绍、游戏流程、胜利条件说明；新手教程能引导玩家完成一局模拟游戏
    - 反向测试：教程未完成时跳过步骤应被提示返回

- AC-2: 多端适配与真人玩家接入
  - 正向测试（预期通过）：
    - 电脑端浏览器访问游戏，能创建房间并作为1名真人玩家参与
    - 手机端浏览器扫码或输入房间号加入，最多3名手机玩家同时在线（总计1-4名真人）
    - 真人玩家在手机端能通过触摸选择目标、投票、使用技能
    - 真人玩家在电脑端能通过鼠标点击完成所有操作
  - 反向测试（预期失败）：
    - 尝试让2名玩家同时从电脑端加入应提示电脑端仅支持1名真人
    - 尝试让5名真人玩家加入应提示超过4人上限
    - 非当前操作玩家尝试投票或使用技能应被阻止并提示"未轮到你操作"

- AC-3: AI玩家系统与多模型支持
  - 正向测试（预期通过）：
    - 在配置界面输入OpenRouter API Token，选择GPT-4、Claude、Gemini、Deepseek、Qwen、Kimi中的任意模型为AI玩家
    - 一局游戏中同时使用6种不同模型的AI玩家，每个AI能根据游戏状态做出决策（投票、使用技能、发言）
    - AI玩家能接收当前游戏信息（自己的身份、已知信息、当前阶段）并返回合法操作
  - 反向测试（预期失败）：
    - 未配置API Token时尝试添加AI玩家应提示需要先配置
    - API Token无效时应提示认证失败
    - AI返回非法操作（如狼人白天杀人）应被系统拒绝并要求重新决策

- AC-4: 语音交互系统
  - 正向测试（预期通过）：
    - 真人玩家在发言阶段通过手机麦克风录音，系统能识别并转为文字显示
    - AI玩家发言时，系统将文字转为语音播放，6个不同AI使用6种不同音色
    - 系统在关键节点语音提示"天黑请闭眼"、"狼人请睁眼"、"狼人请杀人"、"天亮了"等流程引导
    - 所有玩家（真人和AI）能听到其他玩家的发言内容
  - 反向测试（预期失败）：
    - 非发言阶段尝试录音应被阻止
    - 语音识别失败时应提示重新录制
    - 未轮到发言的玩家尝试发言应被拒绝

- AC-5: 游戏流程控制与回合管理
  - 正向测试（预期通过）：
    - 游戏按"天黑-狼人行动-预言家验人-女巫行动-守卫守护-天亮-宣布死亡-遗言-发言-投票-处决"顺序自动推进
    - 每个阶段只有对应角色能操作，其他玩家界面显示等待状态
    - 所有玩家操作完成后自动进入下一阶段，超时未操作则跳过
    - 游戏结束时显示胜负结果和所有玩家身份
  - 反向测试（预期失败）：
    - 狼人阶段非狼人玩家尝试查看狼人聊天应被隐藏
    - 尝试在投票阶段使用夜间技能应被拒绝
    - 已死亡玩家尝试投票或使用技能应被阻止

- AC-6: 电脑端UI与视频录制支持
  - 正向测试（预期通过）：
    - 电脑端显示所有玩家头像、状态（存活/死亡）、当前阶段、倒计时
    - 发言内容以字幕形式实时显示在屏幕上
    - 投票结果、技能使用结果以动画形式清晰展示
    - UI布局适合16:9视频录制，关键信息不被遮挡
  - 反向测试（预期失败）：
    - 隐藏身份信息的玩家视角不应显示其他玩家的真实身份
    - 夜间阶段不应显示其他角色的行动细节

- AC-7: 本地部署与局域网连接
  - 正向测试（预期通过）：
    - 在本地电脑启动服务，生成局域网访问地址和二维码
    - 同一局域网内的手机扫码或输入地址能成功连接
    - 多设备同时在线时游戏数据实时同步
  - 反向测试（预期失败）：
    - 非同一局域网设备尝试连接应超时或提示无法访问
    - 服务未启动时访问地址应显示连接失败

## 路径边界

路径边界定义了实现质量和选择的可接受范围。

### 上界（最大可接受范围）
实现包含完整的6-16人灵活配置系统，支持自定义角色组合和规则变体（如白狼王、守墓人等扩展角色）。AI系统支持6种大模型的高级配置（温度参数、提示词模板自定义）。语音系统集成多种TTS/STT引擎供选择，支持音色微调。UI提供多套主题皮肤和布局模式（观众模式、玩家模式、录制模式）。游戏支持回放系统、数据统计、战绩记录。包含完整的单元测试和集成测试覆盖。

### 下界（最小可接受范围）
实现支持6、9、12、16人四种固定配置，每种配置的角色组合预设且不可修改（仅包含狼人、平民、预言家、女巫、猎人、守卫六种基础角色）。AI系统支持通过OpenRouter接入指定的6种大模型，使用默认参数。语音系统使用一种开源TTS引擎（如Coqui TTS或Edge TTS）和一种STT引擎（如Whisper），为6个AI分配6种不同预设音色。UI采用单一清晰布局，满足视频录制基本需求。游戏流程按标准规则自动推进，支持1-4名真人玩家通过电脑或手机参与。本地部署能在局域网内稳定运行。

### 允许的选择
- 可以使用：
  - 前端框架：React、Vue、Svelte等现代框架
  - 后端框架：Node.js (Express/Fastify/Nest.js)、Python (FastAPI/Flask)、Go等
  - 实时通信：WebSocket、Socket.io、WebRTC
  - 语音技术：Whisper (STT)、Coqui TTS、Edge TTS、Azure TTS、Google TTS等开源或免费方案
  - 数据存储：内存存储、SQLite、JSON文件（本地部署无需复杂数据库）
  - UI库：Tailwind CSS、Material-UI、Ant Design等

- 不可使用：
  - 付费的专有语音服务（除非用户明确提供API密钥）
  - 需要外网访问的云服务（违背本地部署原则）
  - 过时的技术栈（如jQuery、PHP传统架构）

> **确定性设计说明**：草案明确要求支持6种特定大模型（GPT、Claude、Gemini、Deepseek、Qwen、Kimi）、6-16人范围、1-4名真人玩家、本地部署、Web应用形式，这些是固定约束。路径边界在这些约束内提供技术选型和功能深度的灵活性。

### 概念方案

整体架构采用前后端分离的Web应用模式：

```
┌─────────────────────────────────────────────────────┐
│                    客户端（浏览器）                      │
│  ┌──────────────┐  ┌──────────────┐                  │
│  │  电脑端 UI     │  │  手机端 UI     │                 │
│  │  (主视角/录制)  │  │  (玩家操作)    │                 │
│  └──────┬───────┘  └──────┬───────┘                  │
│         └──────┬──────────┘                           │
│                │ WebSocket                             │
└────────────────┼──────────────────────────────────────┘
                 │
┌────────────────┼──────────────────────────────────────┐
│           后端服务器（本地运行）                          │
│  ┌─────────────┴────────────┐                         │
│  │       游戏引擎核心          │                        │
│  │  - 状态机（回合管理）        │                        │
│  │  - 规则验证器               │                        │
│  │  - 玩家管理器               │                        │
│  └──────────┬───────────────┘                         │
│       ┌─────┼──────┐                                  │
│  ┌────┴──┐ ┌┴─────┐ ┌┴──────┐                        │
│  │AI代理  │ │语音系统│ │通知系统 │                       │
│  │管理器  │ │      │ │       │                         │
│  └───┬───┘ └──┬───┘ └───────┘                         │
│      │        │                                        │
│  ┌───┴──────┐ ├── STT (Whisper)                       │
│  │OpenRouter│ └── TTS (多音色引擎)                     │
│  │  API     │                                          │
│  └──────────┘                                          │
└────────────────────────────────────────────────────────┘
```

**游戏引擎**采用有限状态机设计：
- 状态：等待加入 → 分配身份 → 夜间阶段（狼人/预言家/女巫/守卫依次行动）→ 天亮宣布 → 遗言阶段 → 发言阶段 → 投票阶段 → 处决 → 判定胜负 → 循环或结束
- 每个状态有明确的进入/退出条件和超时处理

**AI代理**统一通过OpenRouter API接口调用：
- 为每个AI玩家构建上下文提示词（包含身份、已知信息、当前阶段、可用操作）
- 解析AI返回的结构化决策（JSON格式的行动选择）
- 处理API调用失败的降级策略

**语音系统**双向设计：
- STT（语音转文字）：Whisper或Web Speech API，用于真人玩家发言
- TTS（文字转语音）：Edge TTS或Coqui TTS，为每个AI分配不同音色参数
- 系统主持语音：预录音频或TTS生成的标准提示音

### 相关参考

由于项目从零开始，以下是技术参考：
- OpenRouter API 文档 - 统一的多模型调用接口
- Web Speech API - 浏览器内置语音识别
- Whisper (OpenAI) - 高质量开源语音识别
- Edge TTS / Coqui TTS - 免费多音色语音合成

2. **里程碑 2：网络通信与多端接入**
   - 阶段 A：WebSocket服务搭建（房间管理、事件广播、状态同步）

   - 阶段 C：手机端UI适配（响应式布局、触控优化、扫码加入）

3. **里程碑 3：AI玩家系统**
   - 阶段 A：OpenRouter API集成（统一接口封装、6种模型支持、Token配置界面）
   - 阶段 B：AI决策代理实现（上下文构建、提示词工程、结构化输出解析）
   - 阶段 C：AI行动执行（将AI决策转化为游戏操作、异常处理和重试）

4. **里程碑 4：语音交互系统**
   - 阶段 A：STT集成（真人语音输入识别、实时转文字）

   - 阶段 D：AI语音理解（将其他玩家发言文字传递给AI用于决策）

5. **里程碑 5：规则介绍与教程系统**
   - 步骤 1：规则介绍页面（角色说明、游戏流程、胜利条件、技能规则）

   - 步骤 1：电脑端录制视角优化（16:9布局、字幕显示、动画效果）

### 狼人杀规则要点（基于标准规则研究）

#### 基本规则
- **阵营**：村民阵营（神职+平民）vs 狼人阵营
- **胜利条件**：
  - 狼人胜利：杀光所有神职 或 杀光所有平民（屠边）
  - 好人胜利：白天投票放逐所有狼人

#### 基础角色（6种）
- **狼人**：夜间共同选择杀害一名玩家，允许空刀和自刀
- **平民**：无特殊技能，白天发言和投票
- **预言家**：夜间验证一名玩家是否为狼人
- **女巫**：拥有解药（救人）和毒药（毒杀），各一瓶，解药用完后不再得知被杀者
- **猎人**：被狼人杀或被投票放逐时可开枪带走一人（被毒杀时不能开枪）
- **守卫**：夜间守护一名玩家免受狼人伤害，可自守，不能连续两晚守同一人，同守同救失效

#### 夜间行动顺序
守卫 → 狼人 → 女巫 → 预言家

#### 白天流程
宣布死讯 → 遗言（被杀者最后发言）→ 依序发言 → 投票 → 处决或平安

#### 投票规则
- 存活玩家一人一票
- 票数最多者被放逐
- 平票时进入PK（平票玩家再发言，其他玩家再投票）
- 再次平票则当日平安

#### 推荐配置
- 6人局：2狼2民2神（预言家、女巫）
- 9人局：3狼3民3神（预言家、女巫、猎人）
- 12人局：4狼4民4神（预言家、女巫、猎人、守卫）
- 16人局：5狼6民5神（预言家、女巫、猎人、守卫、白痴）

### 用户确认的关键决策
- **开发方式**：从零开始开发，非移植
- **技术形态**：全Web应用（浏览器访问）
- **游戏主持**：系统自动主持（非真人主持）
- **玩家人数**：6-16人灵活配置（硬性要求）
- **真人玩家**：1-4人（硬性要求），电脑端最多1人，手机端支持多人
- **AI模型**：6种指定大模型全部支持（硬性要求）
- **部署方式**：本地部署，局域网访问

ai也会进行操作，可以api提醒你可以进行操作了，或者你自己决定如何实现这个。同时你要完成agent接入语音系统的功能，比如使用whisper,VibeVoice等项目，这个你可以自己考虑使用现在最新最好的技术，但要支持第一能够听到别的真人或者AI玩家的发言并理解，第二能将自己的发言说出来。每个AI的说话声音要不一样
电脑端的显示UI期望能够清晰明了，优美好看，能够支持我录制视频（需要保证观众能看懂，清晰明了，且看的好玩）
最后游戏需要支持游戏回合引导，比如可以语音提示天黑请闭眼，狼人请睁眼，狼人请杀人等，主持游戏进行，并且主要轮到具体玩家操作时，才能操作，否则不能投票，发言。
